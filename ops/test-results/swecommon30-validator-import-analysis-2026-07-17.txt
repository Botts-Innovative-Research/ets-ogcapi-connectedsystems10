Recommended import pattern
--------------------------
1. Do not import or bundle ets-swecommon30. Import only
   org.opengis.cite:swecommon30-validator.
2. Do not modify TeamEngine source, TeamEngine WAR contents, or TeamEngine
   runtime behavior to make the validator available.
3. Keep all Connected Systems TestNG methods, skip/fail semantics, no-mutation
   policy, and TeamEngine reporting in this ETS.
4. When the validator artifact becomes published, add it as a normal ETS
   dependency and include its jar in the ETS runtime dependency closure that is
   copied into WEB-INF/lib alongside this ETS jar.
5. Exclude validator transitives that are already supplied by the selected
   TeamEngine/ETS runtime, or pin dependencyManagement to the runtime versions
   chosen for the TeamEngine webapp inventory. This avoids adding a second
   competing Jackson/NetworkNT/SLF4J stack.
6. If development must proceed before publication, use a documented local
   install/prebuild step in CI and smoke scripts. A plain POM dependency on
   0.1-SNAPSHOT is not enough for reproducible Docker Maven builds.

Boundary note
-------------
If "do not alter TeamEngine itself" means "do not patch TeamEngine source or the
deployed webapp to integrate SWE Common", the validator import can satisfy that:
the ETS jar and its dependency jar are the integration boundary. If it means
"do not change any TeamEngine container assembly at all", this repository's
existing Dockerfile already applies legacy TeamEngine 5.6.1/JDK 17 runtime
patches unrelated to SWE Common. A SWE Common import should not add new
TeamEngine patches.
