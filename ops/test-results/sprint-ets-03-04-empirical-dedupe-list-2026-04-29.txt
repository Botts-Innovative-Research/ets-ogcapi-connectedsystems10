============================================================================
DOCKER LAYER ANALYSIS (where the 663MB actually comes from)
============================================================================

  Layer                                                         Size
  ──────────────────────────────────────────────────────────────────
  base tomcat:8.5-jre17                                       ~286MB  (immutable)
  apt-get + 3 TE zips download/unzip (RUN)                     25.1MB
  JAXB shared lib download (RUN)                                1.6MB
  COPY --from=builder lib-runtime/ (39 jars)                   39.4MB
  COPY ets-ogcapi-connectedsystems10-0.1-SNAPSHOT.jar           0.3MB
  COPY ets-ctl.zip + unzip                                      2.2MB
  groupadd + useradd + chown -R                                80.2MB  ← DOMINANT cost
  ──────────────────────────────────────────────────────────────────
  Reported total                                              663MB

  KEY INSIGHT: The 80MB chown -R layer is ~12% of the image, caused by Docker's
  copy-on-write semantics — `chown -R tomcat:tomcat /usr/local/tomcat` rewrites
  every file's metadata, materializing them in a new layer. Eliminating this
  requires either (a) chown ONLY the writable dirs (logs, work, temp), or
  (b) building with --chown on each COPY (Dockerfile syntax 1.6 supports this).
  The 39MB lib-runtime COPY is the second-largest discretionary layer.

============================================================================
VERDICT — S-ETS-03-04 outcome
============================================================================

PARTIAL — dedupe applied (4 jars, 1.8MB savings); image cannot reach 550MB
target via the dedupe path alone on the empirical TE 5.6.1 + ETS 0.1-SNAPSHOT
dep tree. The ADR-009 amendment §illustrative table projection of 200-300MB
savings DOES NOT MATCH the actual overlap on this specific TE+ETS combination.
This is exactly the architect-flagged risk GENERATOR-EMPIRICAL-DEDUPE-LIST-
DERIVATION's worst-case symptom and is the empirical justification for
escalating to ADR-009 alternative (b) distroless OR (c) chown-layer-elimination
in Sprint 4.

NEXT-SPRINT RECOMMENDATIONS (Sprint 4):
  1. Eliminate the chown -R layer using --chown=tomcat:tomcat on each COPY
     directive (Dockerfile syntax 1.6 supports this; saves ~80MB).
  2. Investigate intra-WEB-INF/lib duplicate-version dedupe with iterative
     smoke verification (saves additional ~3-4MB but high risk; budget 1 day).
  3. Re-evaluate ADR-009 alternative (b) distroless if (1)+(2) miss target.
